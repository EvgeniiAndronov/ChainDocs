## 🔴 Проблема

### Актуальность

В современном документообороте критически важны:

1. **Неизменяемость** — документы не должны изменяться post-factum
2. **Подтверждение** — несколько сторон должны подтверждать документы
3. **Аудит** — возможность проверить историю изменений
4. **Доверие** — распределённая система без единого центра

### Существующие решения

| Решение | Проблема |
|---------|----------|
| Централизованные БД | Единая точка отказа, недоверие |
| Бумажный документооборот | Медленно, нет аудита |
| Частные блокчейны | Сложно, дорого |

---

## ✅ Решение

### ChainDocs

**Блокчейн-система для документооборота с криптографической подписью**

```
Один документ = Один блок = Неизменяемый хэш
```

### Преимущества

- ✅ **Неизменяемость** — SHA-256 хэш каждого документа
- ✅ **Мульти-подпись** — несколько клиентов подтверждают
- ✅ **Аудит** — полная история в блокчейне
- ✅ **Доверие** — распределённая архитектура

## ⭐ Ключевые особенности

### 1. Динамический консенсус

**Проблема:** При 50 зарегистрированных и 10 активных клиентах требуется 26 подписей (51% от всех), что невозможно.

**Решение:** Расчёт от **активных** ключей (за 24 часа):

```
Зарегистрировано: 50
Активных (24ч):   10
Требуется:        10/2 + 1 = 6 подписей ✅
```

### 2. Activity Tracking

Отслеживание активности каждого ключа:

```go
type KeyActivity struct {
    PublicKey  string    // Публичный ключ
    LastSeen   time.Time // Последняя подпись
    BlockCount int64     // Количество подписей
}
```

### 3. Self-Healing

Автоматическое обнаружение компрометации:

- Детектор чужих подписей
- API отзыва ключей
- Уведомления через вебхуки

### 4. Веб-интерфейс

- Dashboard со статистикой
- Просмотр блоков и подписей
- Загрузка документов
- Управление ключами

## 🎬 Демонстрация

### Сценарий 1: Загрузка документа

```bash
# 1. Загрузить PDF
curl -X POST http://localhost:8080/api/upload \
  -F "file=@contract.pdf"

# Ответ:
{
  "hash": "abc123...",
  "block_hash": "def456...",
  "filename": "contract.pdf"
}
```

### Сценарий 2: Подписание блока

```bash
# Клиент 1
./bin/client -password pass1 -mode oneshot
# [CLIENT] ✅ Block 1 signed successfully!

# Клиент 2
./bin/client -password pass2 -mode oneshot
# [CLIENT] 🎉 CONSENSUS REACHED!
```

### Сценарий 3: Проверка консенсуса

```bash
curl http://localhost:8080/api/blocks/last/consensus | jq

# Ответ:
{
  "signatures": 2,
  "required": 2,
  "percent": 100,
  "consensus_reached": true
}
```

## 🔧 Технические детали

### Криптография

| Алгоритм | Назначение |
|----------|------------|
| **Ed25519** | Подписи блоков |
| **AES-256-GCM** | Шифрование ключей |
| **scrypt** | KDF из пароля |
| **SHA-256** | Хэши документов |

### Хранение

- **BBolt** — embedded key-value БД
- **Бакеты:** blocks, height, pubkeys, activity, revoked
- **Ротация логов** — Lumberjack

### Monitoring

- **Prometheus** — метрики
- **Grafana** — дашборды
- **Health checks** — проверка состояния
