## 📊 Обзор системы

ChainDocs — это **распределённая блокчейн-система** для защиты документов с использованием криптографических подписей нескольких клиентов.

### Ключевые особенности

- ✅ **Блокчейн** — каждый документ создаёт неизменяемый блок
- ✅ **Мульти-подписи** — несколько клиентов подписывают каждый блок
- ✅ **Динамический консенсус** — 51% от активных ключей
- ✅ **Гибридная архитектура** — WebSocket + P2P
- ✅ **Self-healing** — детекция компрометации ключей
- ✅ **Категории документов** — организация по папкам

## 🧩 Компоненты

### 1. Сервер (Coordinator)

**Роль:** Центральный координатор и хранитель блокчейна

**Функции:**
- Хранение блокчейна (BBolt БД)
- Координация консенсуса
- WebSocket Hub для real-time уведомлений
- HTTP API для клиентов и Web UI
- Discovery пиров (список активных клиентов)
- Аудит и логирование

**Порты:**
- `8080` — HTTP API + WebSocket
- `:9001-9003` — не используются (сервер не имеет P2P порта)

**Конфигурация:**
```json
{
  "port": 8080,
  "db_path": "blockchain.db",
  "upload_dir": "./uploads",
  "consensus": {
    "type": "percentage",
    "percentage": 51,
    "min_signatures": 2
  }
}
```

---

### 2. Клиент (Signer)

**Роль:** Независимый подписант блоков

**Функции:**
- Подписание блоков приватным ключом
- Мониторинг новых блоков (WebSocket + polling)
- P2P коммуникация с другими клиентами
- Self-healing детектор чужих подписей
- Graceful shutdown

**Порты:**
- `9001-9003` — P2P сервер для входящих подключений
- Исходящие подключения к другим клиентам

**Режимы работы:**
- **daemon** — постоянная работа с проверкой каждые 5-10 сек
- **oneshot** — однократная проверка и подпись

**Конфигурация:**
```json
{
  "server": "http://localhost:8080",
  "key_file": "client1.enc",
  "password_env": "CHAINDOCS_CLIENT1_PASSWORD",
  "mode": "daemon",
  "daemon": {
    "interval": "5s",
    "sign_unsigned_only": false
  },
  "p2p": {
    "enabled": true,
    "listen_port": 9001
  }
}
```

---

### 3. Keygen (Утилита)

**Роль:** Генерация пар ключей

**Функции:**
- Генерация ключей Ed25519
- Шифрование приватного ключа (AES-256-GCM)
- KDF на основе scrypt

**Использование:**
```bash
./bin/keygen -password "mypassword" -out client.enc
```

## 🔀 Гибридная архитектура

ChainDocs использует **гибридную архитектуру**, сочетающую централизованный сервер для координации и P2P для прямой коммуникации между клиентами.

```
                    ┌─────────────┐
                    │   СЕРВЕР    │
                    │  :8080      │
                    │  • Блокчейн │
                    │  • WebSocket│
                    │  • API      │
                    └──────┬──────┘
                           │ WebSocket (real-time)
         ┌─────────────────┼─────────────────┐
         │                 │                 │
   ┌─────▼──────┐   ┌─────▼──────┐   ┌─────▼──────┐
   │  Клиент 1  │   │  Клиент 2  │   │  Клиент 3  │
   │  P2P:9001  │   │  P2P:9002  │   │  P2P:9003  │
   └─────┬──────┘   └─────┬──────┘   └─────┬──────┘
         │                │                │
         └────────────────┼────────────────┘
                          │
                    P2P Mesh Network
                    (gossip protocol)
```

### Преимущества гибридной архитектуры

| Аспект | Централизованная | P2P | Гибридная |
|--------|-----------------|-----|-----------|
| **Скорость** | 3-10 сек (polling) | Мгновенно | Мгновенно (WS) |
| **Надёжность** | Единая точка отказа | Нет SPOF | Сервер = координация |
| **Аудит** | ✅ Полный | ❌ Сложно | ✅ Полный |
| **Масштабируемость** | Ограничена | Горизонтальная | Средняя |

     │                                  │
     └────────────┬─────────────────────┘
                  │
     ┌────────────▼────────────┐

     │    POST /api/sign       │
     └────────────┬────────────┘
                  ▼
         ┌─────────────┐
         │   Сервер    │

         └─────────────┘

### Сценарий 2: Подключение нового клиента

```
1. Клиент запускается
         │
         ▼
2. Подключение к WebSocket сервера
         │
         ▼
3. Запрос /api/blocks/pending
         │
         ▼
4. Подписание неподписанных блоков
         │
         ▼
5. P2P подключение к другим клиентам
         │
         ▼
6. Обмен ping/pong каждые 30 сек
```

### Сценарий 3: Достижение консенсуса

```
Блок создан (0 подписей)
         │
         ▼
Клиент 1 подписал (1/2 = 50%)
         │
         ▼
Клиент 2 подписал (2/2 = 100%) ✅ КОНСЕНСУС
         │
         ▼
Клиент 3 подписал (3/2 = 100%) ✅ ИЗБЫТОЧНОСТЬ
```

**Формула консенсуса:**
```
required = max(2, ceil(total_keys * 0.51))
consensus_reached = signatures >= required
```

## 🎯 Сценарии работы

### 1. Нормальная работа

**Условие:** Сервер и все клиенты работают

**Поток:**
1. Документ загружается → создаётся блок
2. Сервер рассылает WebSocket уведомление
3. Все 3 клиента подписывают блок
4. Консенсус достигается (3/2)
5. Блок подтверждён

**Время:** < 1 секунды

---

### 2. Частичный отказ клиентов

**Условие:** 1 из 3 клиентов недоступен

**Поток:**
1. Документ загружается → создаётся блок
2. Сервер рассылает WebSocket уведомление
3. 2 клиента подписывают блок
4. Консенсус достигается (2/2)
5. Блок подтверждён

**Время:** < 1 секунды

---

### 3. Отказ сервера

**Условие:** Сервер недоступен

**Поток:**
1. Загрузка документа **невозможна**
2. Существующие блоки **неизменны**
3. Клиенты работают в режиме polling
4. При восстановлении сервера — синхронизация

**Восстановление:** Перезапуск сервера

---

### 4. Позднее подключение клиента

**Условие:** Клиент подключился после создания блоков

**Поток:**
1. Клиент подключается к WebSocket
2. Запрашивает `/api/blocks/pending`
3. Находит неподписанные блоки
4. Подписывает их
5. P2P подключение к другим клиентам

**Время синхронизации:** ~5-10 секунд

## 🔒 Безопасность

### Криптография

| Алгоритм | Назначение | Размер ключа |
|----------|------------|--------------|
| **Ed25519** | Подписи блоков | 256 бит |
| **AES-256-GCM** | Шифрование ключей | 256 бит |
| **scrypt** | KDF из пароля | N=2^15, r=8, p=1 |
| **SHA-256** | Хэши документов | 256 бит |

### Защита ключей

1. **Приватный ключ** хранится зашифрованным
2. **Пароль** не хранится в конфиге
3. **Публичный ключ** регистрируется на сервере
4. **Отозванные ключи** попадают в чёрный список

### Self-Healing

**Детекция компрометации:**
- Клиент проверяет подписи в блоках
- Обнаружение чужой подписи → уведомление
- Webhook для внешних систем
- Auto-revoke (опционально)

**Отзыв ключа:**
```bash
POST /api/revoke
{
  "public_key": "compromised_key",
  "new_public_key": "new_key",
  "new_signature": "signature_by_new_key",
  "reason": "compromised"
}
```
